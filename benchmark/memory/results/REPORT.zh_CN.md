# 长期对话记忆能力评估：基于 LoCoMo 基准的实证研究

## 1. 引言

长期对话记忆是 AI Agent 在多轮会话中与用户交互的核心能力。随着对话不断累积，Agent 需要有效地存储、检索和推理过去的交互内容，以保持连贯且个性化的回复。

本报告使用 **LoCoMo**（Long-Context Conversational Memory）基准评估 trpc-agent-go 的记忆能力，在两种存储后端上对比三种不同的记忆范式，并进一步研究在记忆方案中注入原始对话历史（300 轮和 700 轮）作为额外上下文的效果。

**核心发现：**

- **长上下文基线** 取得最高 F1（0.474），在上下文窗口允许的情况下，完整对话仍是最优方案
- **自动记忆提取**（F1=0.363 pgvector）是最强的记忆方案，达到长上下文基线的 76.7%
- **Agentic 记忆**（F1=0.291 MySQL）显示 LLM 驱动的记忆提取面临信息密度挑战
- **pgvector 在 Auto 模式优于 MySQL**（F1 +0.011），而 Agentic 模式下 MySQL 略优（F1 +0.005）；向量相似度搜索对 Auto 的密集记忆条目更有效
- **注入原始历史降低 F1/BLEU 但提升 LLM Score**，揭示了 Token 级精度与语义质量之间的权衡
- **Token 消耗** 差异显著：长上下文约 18.8K prompt tokens/QA（1 次调用），记忆方案约 2K-9K prompt tokens/QA（2 次调用）

## 2. 方法

### 2.1 基准数据集

使用 **LoCoMo** 数据集（Maharana et al., 2024），包含多个会话对之间的多轮对话：

- 每个样本包含 15-25 个跨数月的对话 session
- 涵盖 5 个类别的 ground-truth QA 对
- 提供 session 级别的 observation 和 summary

**评估规模**：10 个样本，共 1,986 个 QA 对

### 2.2 评估场景

| 场景 | 描述 | 记忆写入 | 记忆读取 |
| --- | --- | --- | --- |
| **Long-Context** | 完整对话作为 LLM 上下文 | 无（全部放入上下文） | 无 |
| **Agentic** | LLM Agent 通过工具调用决定存储内容 | LLM 工具调用 (memory_add) | 记忆检索 |
| **Auto** | 后台提取器自动生成记忆 | 异步提取 | 记忆检索 |

### 2.3 存储后端

| 后端 | 检索方式 | Embedding 模型 |
| --- | --- | --- |
| **pgvector** | 向量相似度（余弦） | text-embedding-3-small |
| **MySQL** | 全文搜索（类 BM25） | 无 |

### 2.4 历史注入实验

在基础场景之外，我们评估将原始对话历史轮次作为上下文消息注入记忆检索结果中：

| 变体 | 描述 |
| --- | --- |
| **无历史** | 仅记忆检索（记忆场景的基线） |
| **+300 轮历史** | 注入最近 300 轮对话作为上下文 |
| **+700 轮历史** | 注入最近 700 轮对话作为上下文 |

历史消息通过 `WithInjectedContextMessages` 注入到 system prompt 之后、session history 之前。

### 2.5 评估指标

与 LoCoMo 论文及业界标准（Mem0, MemMachine）对齐：

| 指标 | 描述 |
| --- | --- |
| **F1 Score** | Token 级别 F1（主要指标） |
| **BLEU Score** | N-gram 重叠精度 |
| **LLM Score** | LLM-as-Judge 语义评估 (0-1) |

### 2.6 QA 分类

| 类别 | 数量 | 描述 |
| --- | --- | --- |
| single-hop | 282 | 单跳问题，来自单个对话片段 |
| multi-hop | 321 | 多跳问题，需要组合多个片段的信息 |
| temporal | 96 | 时间推理问题 |
| open-domain | 841 | 开放域问题，需要世界知识 |
| adversarial | 446 | 对抗性问题，测试鲁棒性（不可回答） |

### 2.7 实验配置

| 参数 | 值 |
| --- | --- |
| 模型 | gpt-4o-mini |
| 评估模型 | gpt-4o-mini |
| 样本数 | 10（完整 LoCoMo-10） |
| 总问题数 | 1,986 |
| LLM 评判 | 启用 |

## 4. 分析

### 4.1 场景对比

#### 4.1.1 长上下文是金标准

长上下文取得最高 F1（0.474），在除对抗性外的所有类别中均最优。这确认了在上下文窗口允许时，提供完整对话文本仍然是最佳方案。但这种方法无法扩展到任意长度的对话历史。

#### 4.1.2 自动提取优于 Agentic

自动记忆提取（F1=0.363）显著优于 Agentic（0.287）。自动提取更系统化——处理所有对话内容，而非依赖 LLM Agent 的选择性工具调用，生成更高密度、语义更丰富的记忆。

#### 4.1.3 对抗鲁棒性与召回能力呈反相关

记忆方案达到较高对抗 F1（0.650-0.825），而长上下文仅 0.663。这是因为基于记忆的方法在未检索到相关记忆时自然返回"信息不可用"，这恰好是对抗性问题的正确答案。而长上下文有完整文本可用时，更容易产生看似合理但实际错误的回答。

### 4.2 历史注入分析

#### 4.2.1 注入历史导致 F1/BLEU 下降

在所有 4 个场景-后端组合中，注入对话历史一致地降低了 F1 和 BLEU 分数：

| 场景 | 后端 | F1（无） | F1（+300） | F1（+700） | 变化 |
| --- | --- | ---: | ---: | ---: | ---: |
| Auto | pgvector | 0.363 | 0.294 | 0.288 | -0.075 |
| Auto | MySQL | 0.352 | 0.282 | 0.290 | -0.062 |
| Agentic | pgvector | 0.287 | 0.272 | 0.274 | -0.013 |
| Agentic | MySQL | 0.291 | 0.271 | 0.278 | -0.013 |

主要原因是**对抗性分数崩溃**。无历史时，对抗性 F1 在 0.650-0.825 之间；注入 +700 轮历史后降至 0.384-0.420。模型在有大量原始对话上下文时，会尝试回答本应拒绝的问题，在对抗性类别（占全部问题的 22%）上损失约 0.25-0.44 F1。

#### 4.2.2 LLM Score 显著提升

虽然 F1/BLEU 下降，但 LLM-as-Judge 分数显著提升：

| 场景 | 后端 | LLM（无） | LLM（+300） | LLM（+700） | 变化 |
| --- | --- | ---: | ---: | ---: | ---: |
| Auto | pgvector | 0.373 | 0.410 | 0.470 | +0.097 |
| Auto | MySQL | 0.373 | 0.397 | 0.477 | +0.104 |
| Agentic | pgvector | 0.280 | 0.365 | 0.459 | +0.179 |
| Agentic | MySQL | 0.285 | 0.368 | 0.463 | +0.178 |

这揭示了一个根本性矛盾：注入的历史使回答在语义上更丰富、更贴合上下文（LLM Score 更高），但也更冗长、与参考答案偏差更大（F1/BLEU 更低）。

#### 4.2.3 开放域问题受益最大

Auto pgvector 的开放域 LLM Score 提升显著：
- 无历史：0.376
- +300：0.532（+41.5%）
- +700：0.690（+83.5%）

这很合理：关于偏好、观点和经历的开放域问题，需要访问离散记忆可能无法捕捉的原始对话细节。

#### 4.2.4 从 300 到 700 轮的边际递减

从 300 到 700 轮的提升相比 0 到 300 是边际性的：
- Auto pgvector LLM Score：0.373 → 0.410（+0.037）→ 0.470（+0.060）
- Auto pgvector F1：0.363 → 0.294（-0.069）→ 0.288（-0.006）

F1 下降在 300 轮后趋于平稳，而 LLM Score 继续线性提升。这表明约 300 轮已捕获了大部分有用的对话上下文，超过后边际收益递减。

### 4.3 Token 消耗分析

#### 4.3.1 长上下文是最耗 Token 的方案

```
Prompt Tokens per QA (No History)

long_context        |==========================================| 18,767
auto_mysql          |====================                      | 9,067
agentic_mysql       |=========                                 | 4,051
agentic_pgvec       |=======                                   | 3,102
auto_pgvec          |====                                      | 1,959
                    +------------------------------------------+
                    0         5,000    10,000   15,000   20,000
```

长上下文每 QA 使用 18,767 prompt tokens（每次发送完整文本），而 Auto pgvector 是最省 Token 的方案，仅 1,959 tokens/QA——**减少 9.6 倍**。

#### 4.3.2 历史注入改变 Token 消耗特征

注入 +700 轮历史后，所有记忆场景的每 QA prompt tokens 增至 21K-23K——接近甚至超过长上下文的 18.8K/QA。这解释了为什么 LLM Score 接近长上下文水平但 F1 下降：模型实际上在做长上下文推理，但带有记忆检索结果的额外噪音。

#### 4.3.3 LLM 调用次数随历史注入减少

纯记忆场景平均 2.0 次调用/QA（一次检索，一次回答）。注入 +700 轮历史后，降至 1.1-1.2 次/QA——模型在注入的上下文中找到答案，更频繁地跳过记忆检索。

#### 4.3.4 MySQL vs pgvector Token 消耗

MySQL 始终消耗更多 prompt tokens（如 Auto：9,067 vs 1,959/QA）。MySQL 的全文搜索返回更长的文本片段，而 pgvector 返回更精炼的向量匹配记忆。尽管 Token 成本更高，MySQL 的 F1 略低，说明更多 Token 不一定意味着更好的答案。

### 4.4 类别分析

#### 4.4.1 时间推理普遍薄弱

时间类问题在所有场景中 F1 最低（0.035-0.103），包括长上下文（0.103）。这表明时间推理对 gpt-4o-mini 来说本质上很困难，与记忆架构无关。

根本原因：
- 对话使用相对时间引用（"去年"、"下个月"），需要根据 session 日期进行解析
- 即使记忆中有明确的 `[DATE:]` 前缀，模型仍难以计算时间关系

#### 4.4.2 多跳推理是持续挑战

多跳 F1 在记忆方案中为 0.091-0.332。长上下文领先（0.332），Auto pgvector 最弱（0.091）。Agentic pgvector（0.142）在多跳上优于 Auto pgvector，表明 Agentic 方法更有选择性的记忆创建可能产生对多跳推理更有用的关联知识。

#### 4.4.3 开放域问题依赖丰富上下文

长上下文在开放域问题中占据主导（F1=0.521），而记忆方案表现较弱（0.129-0.337）。开放域问题通常需要对对话上下文、偏好和态度的细致理解，这些很难用离散的记忆条目捕获。历史注入部分弥补了这一差距（Auto pgvector 开放域 LLM Score：0.376 → 0.690）。

### 4.5 后端对比：pgvector vs MySQL

| 场景 | pgvector F1 | MySQL F1 | 差值 |
| --- | ---: | ---: | ---: |
| Agentic | 0.287 | 0.291 | -0.004 |
| Auto | 0.363 | 0.352 | +0.011 |
| Agentic +700 | 0.274 | 0.278 | -0.004 |
| Auto +700 | 0.288 | 0.290 | -0.002 |

Auto 模式下 pgvector 优于 MySQL 1.1% F1，验证了向量相似度搜索对密集提取记忆的效果。Agentic 模式下 MySQL 略优，可能是因为 Agentic 记忆倾向于更长的文本条目，适合 BM25 风格的关键词匹配。**注入历史后，后端差异消失**——注入的对话上下文主导了检索质量。

### 4.6 方差分析

各样本 F1 分数显示明显方差：
- **长上下文**：0.436 - 0.532（范围 0.096），相对稳定
- **Auto pgvector**：0.257 - 0.442（范围 0.185），方差较大
- **Agentic pgvector**：0.245 - 0.342（范围 0.097），方差中等

Auto 提取的高方差表明，某些对话结构本质上更难进行记忆提取和检索。话题交错更复杂的样本（如 locomo10_6）得分往往更低。

### 5.3 分析

**trpc-agent-go 在记忆框架横向对比中的定位：**

1. **Multi-Hop 推理领先**：trpc-agent (LC) 的 multi-hop F1（0.332）**位列所有参评框架第一**，超越 Mem0（0.286）、LangMem（0.260）等专用记忆系统。这一结果表明，在需要跨 session 组合多条事实的复杂推理场景中，trpc-agent-go 的架构设计具有显著优势

2. **Open-Domain 检索第一**：trpc-agent (LC) 在开放域类别取得 0.521 的最高 F1，领先 Zep（0.496）、Mem0g（0.493）等框架。开放域问题涵盖偏好、态度、生活经历等细粒度信息，该结果体现了框架在细致语义理解方面的能力

3. **Single-Hop 保持竞争力**：trpc-agent (LC) 的 single-hop F1（0.324）与 OpenAI Memory（0.343）处于同一梯队，表明基础事实检索能力具备良好水准

4. **Overall J 与 A-Mem 持平**：Long-Context 模式的 Overall J（0.487）与 A-Mem（0.484）相当。需要注意的是，Mem0 论文的 LLM Judge 评估 10 次取均值，而本工作 Judge 仅评估 1 次，评估次数的差异对分数稳定性有系统性影响

5. **Temporal 类别有优化空间**：当前各模式的 temporal F1 尚处于较低水平（< 0.1），相比 Mem0g（0.516）存在明显差距。时间推理需要对对话中的相对时间表达（"去年""下个月"）进行精确解析，这是后续版本的重点优化方向。值得注意的是，OpenAI Memory（0.140）在该类别同样表现有限，说明时间推理对多数框架而言均具有挑战性

6. **对抗鲁棒性是独特优势**：虽然 Mem0 论文未纳入 adversarial 类别，但本工作的 adversarial F1 达到 0.650–0.825（详见第 4 章），记忆方案在识别不可回答问题上表现出色。这一能力对生产环境中的安全性和可靠性至关重要，也是 trpc-agent-go 区别于纯检索类框架的重要特征

7. **Auto/Agentic 模式的发展潜力**：作为通用 Agent 框架的记忆子系统，Auto 和 Agentic 模式的当前表现反映了首个评估版本的基线水平。相比 Mem0 等以记忆为核心产品的专用系统，trpc-agent-go 的记忆模块在保持框架整体通用性的同时，已在 multi-hop 和 open-domain 等关键类别展现竞争力。随着时间索引、图结构记忆等特性的引入，预期将有显著提升空间

> **可比性说明：**
> - 两者均使用 LoCoMo 基准（10 个对话）和 GPT-4o-mini 推理；本工作的 LLM Judge 仅评估 1 次，Mem0 论文评估 10 次取均值，后者的结果方差更小、分数更稳定
> - Mem0 等框架为专用记忆系统，通过 API 直接搜索+回答；本工作的 Auto/Agentic 模式通过 Agent 工具调用链路实现，架构层次更多
> - 本工作的延迟为包含完整 Agent 工具调用链路的端到端延迟，不直接等同于纯记忆检索延迟

## 6. 讨论

### 6.1 当前实现的优势

1. **日期感知记忆服务**：`datePrefixMemoryService` 自动为 Agentic 记忆注入 `[DATE: ...]` 前缀，无需依赖 LLM 遵从指令即可改善时间推理
2. **强对抗鲁棒性**：记忆方案达到 0.65-0.83 的对抗 F1，能正确识别不可回答问题
3. **自动提取方案**：后台提取器在记忆质量和系统复杂度之间提供了良好平衡
4. **Token 效率**：Auto pgvector 以仅 10.4% 的 prompt tokens 达到长上下文 76.7% 的 F1

### 6.2 历史注入的权衡

历史注入实验揭示了记忆系统设计的关键洞察：

- **纯记忆**（无历史）：最佳 F1/BLEU，最佳对抗鲁棒性，但语义质量较低
- **记忆 + 历史**：最佳 LLM Score 和开放域表现，但精确指标和对抗鲁棒性下降

这表明**混合方案**适合生产环境：将记忆检索作为主要上下文来源，在检测到开放域或需要细致理解的问题时，选择性地注入相关历史片段（而非全部 300-700 轮）。

### 6.3 Token 成本考量

| 方案 | Prompt Tokens/QA | 相对成本 | F1 | 质量权衡 |
| --- | ---: | ---: | ---: | --- |
| Long-Context | 18,767 | 1.00x | 0.474 | 最高质量，最高成本 |
| Auto pgvector | 1,959 | 0.10x | 0.363 | 最佳性价比 |
| Auto MySQL | 9,067 | 0.48x | 0.352 | 中等成本，无需 embedding |
| Auto pgvec +700 | 21,445 | 1.14x | 0.288 | 成本高于 LC，F1 更低 |

Auto pgvector 提供最优质量-成本权衡：以仅 10.4% 的 Token 成本达到长上下文 76.7% 的 F1。+700 轮历史注入从成本角度来看是不合算的——Token 消耗超过长上下文，但 F1 更低。

### 6.4 局限性与未来工作

1. **时间推理仍然薄弱**（所有场景 F1 < 0.1），后续应探索专用时间索引和推理模块
2. **多跳推理差距**：记忆方案难以跨 session 组合事实，图结构记忆可能有帮助
3. **开放域性能**：记忆压缩不可避免地丢失细节，分层记忆（摘要 + 细节）可能弥补此差距
4. **模型能力上限**：gpt-4o-mini 的提取和推理能力限制了所有场景，更强的模型（如 GPT-4o、Claude）可能显著改善结果
5. **选择性历史注入**：不注入全部轮次，而是使用相关性过滤只注入相关的对话片段

### 6.5 生产使用建议

| 使用场景 | 推荐方案 |
| --- | --- |
| 短对话历史 (< 50K tokens) | 长上下文（无需记忆） |
| 长期运行的 Agent（数月历史） | 自动提取 + pgvector |
| 语义质量优先 | 记忆 + 选择性历史注入 |
| 低延迟要求 | Agentic + MySQL |
| 成本敏感部署 | Auto pgvector（节省 10 倍 Token） |

---

## 7. 结论

本评估证明 trpc-agent-go 的记忆系统在多种范式下均能提供有效的长期对话记忆。自动提取 + pgvector 方案取得了最佳的召回与鲁棒性平衡，达到长上下文基线 F1 的 76.7%，同时保持强对抗鲁棒性（0.771），并且仅消耗 10.4% 的 prompt tokens。

历史注入实验揭示了一个重要权衡：注入原始对话历史提升了语义质量（LLM Score +0.10~0.18），但降低了 Token 级精度（F1 -0.01~0.08）和对抗鲁棒性。这确认了**结构化记忆提取在事实召回任务上比暴力上下文注入更有效**，而历史注入对开放域、需要细致理解的问题有附加价值。

Token 消耗分析显示 Auto pgvector 是最具性价比的方案，每 QA 仅使用 1,959 prompt tokens，而长上下文为 18,767。+700 轮历史注入后 Token 成本超过长上下文水平（21K+/QA），但 F1 更低，使全量历史注入在实践中不合算。

核心结论：
1. **Auto > Agentic**：在 gpt-4o-mini 下的记忆方案有效性排序
2. **pgvector > MySQL**：Auto 模式下更优；Agentic 模式下 MySQL 略优
3. **时间推理** 是所有方案的主要瓶颈
4. **对抗鲁棒性** 是记忆系统的天然优势
5. **历史注入** 提升语义质量但降低精度——推荐混合选择性方案
6. **Token 效率**：Auto pgvector 以 10.4% 的 Token 成本达到长上下文 76.7% 的 F1
