# PromptIter Tool Description Example

This example runs PromptIter end to end through `engine.Run` against a single `llmagent` travel-operations candidate.

The candidate starts with one intentionally weak tool description:

```text
lookup_record: Look up a traveler loyalty-profile record.
```

PromptIter optimizes only the candidate tool surface generated by
`structure.SurfaceID(candidateAgentName, structure.SurfaceTypeTool, "lookup_record")`, which resolves to:

```text
candidate#tool.lookup_record
```

## Data Files

The example loads these files from `./data/promptiter-flight-tool-app/` by default:

```text
flight-tool-train.evalset.json
flight-tool-validation.evalset.json
flight-tool.metrics.json
```

The train and validation evalsets contain short flight-status and delay questions. Each case stores the expected tool call name, arguments, deterministic local tool result, and final response. The tool schema uses neutral record fields such as `state`, `minutes`, `location`, `scheduled`, and `updated`, so the optimized description has to explain that this record lookup is a flight-status lookup. The shared metric file combines exact `tool_trajectory_avg_score` matching with a small `llm_rubric_critic` that checks whether the final answer uses the status, delay, gate, and time fields returned by the tool.

The intended optimization is to replace the misleading loyalty-profile declaration with a flight lookup declaration. The train and validation cases use plain flight numbers as tool arguments, so the example measures whether the tool description makes the tool selection and final answer grounding clearer instead of asking PromptIter to infer hidden query-key prefixes. The validation set still contains a held-out gate-only question, so the example tests generalization without assuming the small dataset should reach a perfect score.

## Environment Variables

| Variable | Description | Default |
| --- | --- | --- |
| `OPENAI_API_KEY` | API key for the OpenAI-compatible endpoint | `` |
| `OPENAI_BASE_URL` | Base URL for the OpenAI-compatible endpoint | `https://api.openai.com/v1` |

## Flags

| Flag | Description | Default |
| --- | --- | --- |
| `-data-dir` | Directory containing evaluation set and metric files | `./data` |
| `-output-dir` | Directory where evaluation results will be stored | `./output` |
| `-model` | Model identifier used by the candidate travel agent | `deepseek-v3.2` |
| `-judge-model` | Model identifier used by the judge agent | `gpt-5.2` |
| `-worker-model` | Model identifier used by the PromptIter worker agents | `gpt-5.2` |
| `-max-rounds` | Maximum PromptIter optimization rounds | `4` |
| `-eval-case-parallelism` | Maximum number of eval cases processed in parallel | `8` |
| `-parallel-inference` | Enable parallel inference across eval cases | `true` |
| `-parallel-evaluation` | Enable parallel evaluation across eval cases | `true` |
| `-backward-case-parallelism` | Maximum number of train eval cases processed in parallel during backward; `0` uses `GOMAXPROCS` | `8` |
| `-parallel-backward` | Enable parallel backward processing across train eval cases | `true` |
| `-surface-parallelism` | Maximum number of target surfaces processed in parallel; `0` uses `GOMAXPROCS` | `8` |
| `-parallel-surface-stages` | Enable parallel aggregation and optimization across target surfaces | `true` |
| `-min-score-gain` | Minimum validation score gain required to accept a patch | `0.01` |
| `-max-rounds-without-acceptance` | Maximum consecutive rejected rounds before stopping | `3` |
| `-target-score` | Target validation score that stops optimization when reached | `1.0` |

## Run

```bash
cd examples/evaluation/promptiter/tooldesc
export OPENAI_BASE_URL="https://your-openai-compatible-endpoint/v1"
export OPENAI_API_KEY="your-api-key"
go run . \
  -model "deepseek-v3.2" \
  -judge-model "gpt-5.2" \
  -worker-model "gpt-5.2"
```

## Sample Iteration Result

One sample run with the default flags improved validation score from `0.42` to `0.96`: round 1 and round 2 were accepted, while round 3 and round 4 were rejected because they did not improve over the current accepted declaration.

```text
Data directory: ./data
Result directory: ./output
Target surface ID: candidate#tool.lookup_record
Initial tool declarations:
  lookup_record: "Look up a traveler loyalty-profile record."
Accepted tool declarations:
  lookup_record: "Look up an operational record by key and return the full current status details, including state, delay minutes, location/gate code, and scheduled vs updated (estimated) local times."
Initial validation score: 0.42
Final accepted validation score: 0.96
Rounds executed: 4
Round 1 -> train 0.50, validation 0.92, accepted true, delta 0.50, stop=false (continue optimization)
Round 2 -> train 0.88, validation 0.96, accepted true, delta 0.04, stop=false (continue optimization)
Round 3 -> train 1.00, validation 0.96, accepted false, delta 0.00, stop=false (continue optimization)
Round 4 -> train 0.88, validation 0.96, accepted false, delta 0.00, stop=true (max rounds reached)
```

Round 1 fixes the misleading loyalty-profile declaration by describing the tool as an operational status lookup. Round 2 adds the fields that matter for the held-out cases, especially location/gate and scheduled versus updated estimated times. Later rounds do not improve validation score, so PromptIter keeps the round 2 declaration.

## What It Does

- Loads fixed train and validation tool-trajectory evalsets from local files.
- Runs the candidate with one local function tool: `lookup_record`.
- Evaluates exact tool call name, arguments, and result through `tool_trajectory_avg_score`.
- Evaluates final-answer use of tool result fields through a lightweight `llm_rubric_critic`.
- Targets only the generated `candidate#tool.lookup_record` surface, leaving the candidate instruction unchanged.
- Applies accepted patches as declaration overrides, so tool execution still goes through the original Go functions.
- Prints the initial tool declarations, accepted declarations, score changes, and per-round tool patches.
- Writes raw evaluation results under `./output`.
