# Design

The example is deliberately small, but it preserves the production path that matters for auditability. Fake mode builds a candidate `llmagent`, runs it through `runner.NewRunner`, evaluates it with `evaluation.AgentEvaluator`, and lets the PromptIter engine describe, patch, accept, and re-evaluate real agent surfaces. The deterministic model and fake PromptIter workers are only test doubles around the external LLM calls. Surface changes still flow through the existing profile compiler and `agent.WithSurfacePatchForNode`, so the report can point to concrete surface IDs and candidate profiles instead of undocumented prompt strings.

Failure attribution is derived from evaluation artifacts rather than optimizer intent. The report inspects metric names, failed metric reasons, expected and actual invocations, tool calls, tool arguments, final responses, and the terminal execution trace step. That keeps categories such as `tool_not_called`, `tool_arguments_mismatch`, `final_response_mismatch`, and `route_error` reproducible. Deterministic metrics always emit a non-empty reason on failure, which also protects PromptIter's loss extraction path.

PromptIter acceptance and the final gate have different jobs. PromptIter acceptance decides whether a round's patch becomes the current accepted profile based on score gain. The final gate decides whether that accepted profile is safe to publish. The gate only reads validation delta, critical case configuration, cost, and latency. This separation lets the example demonstrate overfitting: the second round can improve aggregate validation enough to be accepted by PromptIter, while still being rejected for introducing a new hard fail on a critical validation case.

Trace smoke is intentionally outside the optimization loop. Trace-mode evalsets replay recorded `actualConversation` and `executionTrace` data, so they can verify evaluation compatibility, report adaptation, and attribution, but they cannot prove that a prompt patch changes future candidate inference. For that reason the trace report sets a fixed skipped-optimization reason. Both modes write structured JSON and Markdown reports with prompt hash, target surfaces, rounds, deltas, gate reasons, attribution, cost, latency, and trace-smoke status so each decision can be reviewed later.
