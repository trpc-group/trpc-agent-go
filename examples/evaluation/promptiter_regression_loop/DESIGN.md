# Design

The regression loop keeps optimization evidence separate from release authority. The Evaluation Service runs the baseline and every candidate with run details and execution tracing enabled. A normalization layer converts framework results into an ordered, report-facing representation and rejects duplicate or missing case and metric identities, non-finite scores, and threshold changes. This prevents an incomplete result from looking like an unchanged result. Usage is marked measured only when all contributing traces provide measurement evidence.

Failure attribution is deterministic and evidence-driven. Metric configuration identifies strong signal families such as final response, tool trajectory, route, format, and knowledge retrieval. Metric reasons can refine a tool failure into tool selection or tool argument failure. Execution errors and failed traces take precedence. The classifier deliberately does not search generated final-response text for keywords, because a correct answer can discuss a “wrong tool” or “route” without proving that the agent made that execution mistake. Unknown business metrics receive an explicit metric-failure fallback, so every failed case remains explainable without inventing a diagnosis.

PromptIter is connected through its real engine, candidate agent structure, backwarder, aggregator, and optimizer contracts. Each external attempt runs one PromptIter round. Its internal acceptance is recorded but does not authorize production writeback. After PromptIter proposes a profile, the pipeline independently evaluates that exact profile on train and validation, computes case and metric deltas, and submits the validation evidence to the release gate.

The gate compares a candidate with both the original baseline and the last accepted baseline. The report preserves both views: `delta` is the accepted-baseline comparison used for the current release decision, while `baselineDelta` is the immutable original-baseline comparison used for audit and cumulative context. It requires configurable score gain, rejects new failures, protects named critical cases, and enforces measured token or call budgets. This dual comparison prevents cumulative regression, while an independent validation set rejects a candidate that improves training through overfitting. Later rejected candidates never replace an earlier accepted candidate.

Audit output is versioned JSON plus human-readable Markdown. It records prompts, patches, evaluations, deltas, attribution, gate reasons, seed, model and configuration identity, usage, and latency. Fake mode reports measured tokens and calls as its cost basis rather than fabricating currency pricing. Both files are written and synced in a staging directory, then published with one directory rename. Existing run generations are never overwritten, making each decision reproducible and reviewable before any separate source-prompt writeback step.

A second integration test targets a tool-description surface with the real engine and Evaluation Service. It proves the evidence, gate, and report contracts are surface-neutral, while also exposing the remaining abstraction boundary: generic profile-to-runtime compilation is engine-internal. Until external candidate re-evaluation has a stable public cross-surface contract, the complete orchestration should remain an example rather than a public framework API.
